# Capítulo 4 — Autenticación y Autorización

Este capítulo detalla la implementación de la capa de autenticación y autorización de TaskManager API y la compara con los distintos flujos del estándar **OAuth 2.0**, tal y como solicita el enunciado.

## 4.1. Conceptos previos: autenticación vs autorización

Aunque a menudo se usan como sinónimos, son dos conceptos distintos:

- **Autenticación** (*authentication*): verificar **quién es** el usuario. Responde a la pregunta *"¿eres quien dices ser?"*. El mecanismo típico es la comprobación de credenciales (email + contraseña).
- **Autorización** (*authorization*): verificar **qué puede hacer** un usuario ya autenticado. Responde a la pregunta *"¿tienes permiso para hacer esta operación sobre este recurso?"*. En TaskManager se traduce en: *"¿este proyecto te pertenece?"*.

La API implementa ambas capas: autenticación en `/auth/login` (devuelve un token) y autorización en cada endpoint protegido (verifica el token y filtra los recursos por propietario).

## 4.2. Mecanismo implementado: JWT

TaskManager utiliza **JSON Web Tokens (JWT)** como mecanismo de autenticación stateless. El flujo es:

1. El cliente envía credenciales (email + contraseña) a `POST /auth/login`.
2. El servidor verifica las credenciales contra la base de datos:
   - Busca al usuario por email.
   - Compara la contraseña enviada con el hash almacenado usando `bcrypt.compareSync()`.
3. Si las credenciales son válidas, el servidor genera un **JWT firmado** con su clave secreta y lo devuelve.
4. El cliente almacena el token y lo envía en cada petición posterior en la cabecera `Authorization: Bearer <token>`.
5. El servidor verifica el token en cada petición protegida (middleware) y extrae el `userId` del payload.

### 4.2.1. Hash de contraseñas con bcrypt

Las contraseñas **nunca** se almacenan en texto plano. Se utiliza el algoritmo `bcrypt` con factor de coste 10:

```javascript
// Registro
const hashedPassword = bcrypt.hashSync(password, 10);
db.query("INSERT INTO users (..., password) VALUES (..., ?)", [..., hashedPassword]);

// Login
const validPassword = bcrypt.compareSync(password, user.password);
```

`bcrypt` añade automáticamente un *salt* aleatorio al hash, por lo que dos contraseñas idénticas producen hashes distintos. Esto protege frente a ataques de tablas rainbow y dificulta los ataques por fuerza bruta.

### 4.2.2. Estructura del token JWT

Un JWT está compuesto por tres partes separadas por puntos, codificadas en Base64URL:


El **payload** que se firma en TaskManager contiene:

```javascript
{
  id: user.id,
  username: user.username,
  email: user.email,
  iat: <timestamp de emisión>,
  exp: <timestamp de expiración: 1 hora después>
}
```

La generación:

```javascript
const token = jwt.sign(
    { id: user.id, username: user.username, email: user.email },
    process.env.JWT_SECRET,
    { expiresIn: "1h" }
);
```

La **firma** se calcula como `HMAC-SHA256(base64(header) + "." + base64(payload), JWT_SECRET)`. Sin conocer `JWT_SECRET`, es computacionalmente inviable falsificar un token, ya que cualquier modificación del payload invalida la firma.

### 4.2.3. Verificación del token (middleware)

El middleware `auth.middleware.js` se encarga de validar el token en cada petición protegida:

```javascript
function authMiddleware(req, res, next) {
    const authHeader = req.headers["authorization"];

    if (!authHeader) {
        res.status(401).json({ message: "Token no proporcionado" });
        return;
    }

    const token = authHeader.split(" ")[1];

    jwt.verify(token, process.env.JWT_SECRET, function(err, user) {
        if (err) {
            res.status(403).json({ message: "Token no válido" });
            return;
        }
        req.user = user;
        console.log("Usuario autenticado:", req.user);
        next();
    });
}
```

`jwt.verify()` realiza tres comprobaciones:

1. Que la firma sea válida (token no manipulado).
2. Que el token no haya expirado (`exp` no superado).
3. Que el algoritmo de firma coincida con el esperado.

Si todas pasan, el payload se adjunta a `req.user`, queda disponible para los controladores y la petición continúa.

## 4.3. Control de acceso por propietario

La autenticación garantiza saber **quién** hace la petición. La autorización determina **qué** puede hacer. En TaskManager, el control de acceso se aplica a nivel de **propietario del recurso**:

- En `projects`: el filtro `WHERE user_id = ?` con el id del token impide acceder a proyectos ajenos.
- En `tasks`: el `JOIN` con `projects` filtrando por `user_id` impide acceder a tareas de proyectos ajenos.
- En `users/me`: el id se extrae del token, no del path, por lo que es imposible modificar a otro usuario.

Este es el cumplimiento del requisito valorado del enunciado: *"O utilizador autenticado apenas consegue aceder aos seus próprios recursos"*.

## 4.4. Comparación con los flujos de OAuth 2.0

OAuth 2.0 es un **estándar de autorización** definido en la [RFC 6749](https://datatracker.ietf.org/doc/html/rfc6749). No es un protocolo de autenticación, sino un marco para delegar acceso a recursos protegidos. Define varios *flows* (también llamados *grant types*), cada uno pensado para un caso de uso específico.

A continuación se comparan los principales flujos con la implementación actual de TaskManager.

### 4.4.1. Roles de OAuth 2.0

OAuth 2.0 define cuatro roles:

| Rol | Definición | Equivalente en TaskManager |
|---|---|---|
| **Resource Owner** | El usuario final, propietario de los datos | El usuario `demo1`, `demo2`, etc. |
| **Resource Server** | Servidor que aloja los recursos protegidos | La API TaskManager |
| **Authorization Server** | Servidor que emite tokens | El propio TaskManager (no está separado) |
| **Client** | Aplicación que accede a los recursos | Postman, Swagger UI, frontend |

En OAuth 2.0 estricto, el Authorization Server y el Resource Server son entidades **separadas**. En TaskManager, ambos roles los desempeña la misma aplicación, lo que simplifica la arquitectura pero limita la federación con terceros.

### 4.4.2. Authorization Code Flow

Es el flujo más utilizado y seguro de OAuth 2.0. Pensado para aplicaciones web con backend.

**Pasos del flujo:**

1. El usuario hace clic en *"Iniciar sesión con Google"*.
2. El cliente redirige al usuario al Authorization Server (Google).
3. El usuario se autentica en Google e **autoriza** al cliente.
4. Google redirige al cliente con un *authorization code* temporal.
5. El cliente intercambia ese code (junto con su `client_id` y `client_secret`) por un **access token**.
6. El cliente usa el access token para acceder a la API.

**Diferencias con TaskManager:**

| Aspecto | Authorization Code Flow | TaskManager (JWT) |
|---|---|---|
| Quién autentica | Tercero (Google, GitHub, etc.) | El propio servidor TaskManager |
| Almacena contraseñas | No (delega en el tercero) | Sí (con bcrypt) |
| Número de pasos | 6 (incluye redirecciones) | 2 (login → token) |
| Adecuado para | Delegar identidad a un tercero | Aplicación con cuentas propias |
| Complejidad | Alta | Baja |

Este flujo sería el adecuado si TaskManager quisiera ofrecer *"login con Google"* en lugar de gestionar contraseñas propias. Es el flow recomendado para producción cuando interesa delegar la gestión de identidad.

### 4.4.3. Resource Owner Password Credentials Flow

Este flujo permite que el cliente intercambie directamente el email y la contraseña del usuario por un token. **Es el más parecido al implementado en TaskManager**.

**Pasos del flujo:**

1. El cliente solicita las credenciales al usuario.
2. El cliente envía las credenciales al Authorization Server.
3. El Authorization Server las verifica y emite un access token.

**Comparación con TaskManager:**

| Aspecto | Password Credentials Flow | TaskManager (JWT) |
|---|---|---|
| Envío de credenciales | Sí, al Authorization Server | Sí, a `/auth/login` |
| Devuelve token | Sí (access token, opcionalmente refresh token) | Sí (JWT) |
| Token autocontenido | No necesariamente | Sí (JWT contiene el payload) |
| Refresh token | Sí | No implementado |

La implementación de TaskManager es **conceptualmente equivalente** a este flow, con las siguientes diferencias técnicas:

- TaskManager **no implementa refresh tokens**. Cuando el JWT expira (1 hora), el usuario debe hacer login de nuevo. En un escenario real se añadiría un endpoint `/auth/refresh` que devolviera un nuevo access token a partir de un refresh token de larga duración.
- El token de TaskManager es un **JWT autocontenido** (stateless), mientras que en OAuth 2.0 clásico el token puede ser opaco y requerir consulta al Authorization Server (introspection) para validarlo.

> **Nota**: el propio estándar OAuth 2.0 desaconseja este flujo (RFC 6749 §1.3.3) porque obliga al cliente a manejar credenciales en texto plano, lo que rompe el principio de delegación. Actualmente se considera *legacy*. Sin embargo, sigue siendo el modelo más común en APIs con cuentas propias y sin federación, como es el caso de TaskManager en este M1.

### 4.4.4. Client Credentials Flow

Este flujo está pensado para comunicación **máquina a máquina**, sin usuario final involucrado.

**Pasos del flujo:**

1. El cliente (otro servicio backend) envía sus propias credenciales (`client_id` + `client_secret`).
2. El Authorization Server emite un access token asociado al cliente, no a un usuario.

**Comparación con TaskManager:**

No es aplicable al caso de uso. TaskManager está diseñado para usuarios humanos finales, no para integraciones entre servicios. Si en el futuro se quisiera permitir que otra aplicación accediera a TaskManager *en nombre propio* (por ejemplo, para tareas de mantenimiento o sincronización), este sería el flow indicado.

### 4.4.5. Implicit Flow (obsoleto)

Pensado originalmente para aplicaciones SPA (single-page applications) sin backend, donde el token se devolvía directamente en la URL.

**Estado actual**: la [OAuth 2.0 Security Best Current Practice (RFC 9700)](https://datatracker.ietf.org/doc/html/rfc9700) lo declara **obsoleto** por riesgos de fuga del token (queda registrado en historial del navegador, logs, etc.). Para SPAs hoy se recomienda **Authorization Code + PKCE**.

No es aplicable a TaskManager y no se considera una alternativa válida.

### 4.4.6. Authorization Code + PKCE

Variante del Authorization Code Flow diseñada para clientes públicos (móviles, SPAs) que no pueden almacenar de forma segura un `client_secret`.

**Diferencia clave con Authorization Code**: en lugar de un `client_secret`, el cliente genera un *code verifier* aleatorio y envía su hash (*code challenge*) en la petición inicial. Al canjear el code por el token, debe presentar el *code verifier* original, demostrando que es el mismo cliente que inició el flow.

**Comparación con TaskManager**: si TaskManager evolucionara a tener un frontend SPA (React, Vue) y se integrara con un proveedor externo de identidad (Google, Auth0), este sería el flow recomendado actualmente.

## 4.5. Tabla resumen comparativa

| Flow OAuth 2.0 | Caso de uso típico | Aplicable a TaskManager | Por qué |
|---|---|---|---|
| **Authorization Code** | Web con backend, login federado | Como mejora futura | Si se quisiera *"login con Google"* |
| **Authorization Code + PKCE** | SPAs, apps móviles | Como mejora futura | Si se hiciera frontend SPA |
| **Resource Owner Password Credentials** | API con cuentas propias | **El más cercano al actual** | Es lo que esencialmente hace `/auth/login` |
| **Client Credentials** | Comunicación máquina-máquina | No aplicable | No hay clientes no-humanos |
| **Implicit** | (Obsoleto) | No aplicable | Inseguro, deprecado por la RFC 9700 |

## 4.6. Justificación de la elección actual y mejoras futuras

Para el alcance del M1, la implementación con JWT + bcrypt es:

- **Suficiente** para cubrir todos los requisitos obligatorios del enunciado (autenticación, autorización, control por propietario, log del usuario).
- **Simple**: no requiere infraestructura adicional (Authorization Server externo, almacenamiento de tokens en BD, etc.).
- **Conceptualmente alineada** con el flow *Resource Owner Password Credentials* de OAuth 2.0.

Como **mejoras futuras**, fuera del alcance del M1, se podrían implementar:

1. **Refresh tokens**: para evitar que el usuario tenga que hacer login cada hora. Se almacenarían en la base de datos con su `user_id` y fecha de expiración (típicamente días o semanas).
2. **OAuth 2.0 con proveedor externo**: integrar Passport.js + estrategia de Google/GitHub para ofrecer login federado siguiendo el Authorization Code Flow.
3. **Roles y permisos**: ampliar el payload del token con un campo `role` (admin, user, etc.) y un middleware adicional de autorización por rol.
4. **Revocación de tokens**: en JWT stateless no es posible invalidar tokens antes de su expiración. Para soportarlo, se introduciría una lista negra (`blacklist`) en Redis o se migraría a tokens opacos con introspección.
5. **Rotación de claves**: rotar `JWT_SECRET` periódicamente con sistemas de key versioning.

## 4.7. Conclusión

La capa de autenticación y autorización implementada en TaskManager cubre los requisitos del enunciado mediante un esquema **JWT + bcrypt** sencillo, robusto y bien establecido en la industria. Conceptualmente, se sitúa próxima al flow *Resource Owner Password Credentials* de OAuth 2.0.

El uso de JWT stateless permite escalar horizontalmente la aplicación sin necesidad de almacenar sesiones, y la verificación del token en cada petición protegida (mediante middleware) garantiza tanto la autenticación como el log de auditoría que el enunciado solicita.

La adopción de un Authorization Server externo (Google, Auth0, etc.) mediante Authorization Code + PKCE sería el siguiente paso natural si el proyecto evolucionara hacia un entorno de producción con federación de identidad.
