# Capítulo 2 — Arquitectura y modelo de datos

## 2.1. Arquitectura general

La aplicación sigue una **arquitectura de capas** desplegada en **dos contenedores Docker** orquestados con Docker Compose. La comunicación entre componentes es la siguiente:

| Componente | Tipo | Comunicación |
|---|---|---|
| **Cliente** (Postman, navegador, etc.) | Externo | Envía peticiones HTTP con JSON y cabecera `Authorization: Bearer <JWT>` al contenedor de la API |
| **`taskmanager-app`** | Contenedor Node.js 20 + Express 5 | Recibe peticiones HTTP en el puerto 3000, las procesa y consulta MySQL |
| **`taskmanager-mysql`** | Contenedor MySQL 8.0 | Recibe consultas SQL del contenedor de la API por el puerto 3306 (interno) |
| **Red `taskmanager-net`** | Red interna de Docker | Permite que los contenedores se comuniquen entre sí usando nombres de servicio (`mysql:3306`) en lugar de IPs |
| **Volumen `mysql_data`** | Volumen Docker | Garantiza la persistencia de los datos aunque el contenedor de MySQL se reinicie |

El cliente externo solo accede al puerto 3000 de la API. MySQL queda aislado dentro de la red de Docker (aunque se expone el puerto 3307 en el host para facilitar inspección durante el desarrollo).

## 2.2. Estructura del código (capas)

Dentro del contenedor de la aplicación, el código está dividido en cuatro capas con responsabilidades claras:

| Capa | Carpeta | Responsabilidad |
|---|---|---|
| **Routes** | `src/routes/` | Definen los endpoints HTTP y los enlazan con los controladores. |
| **Middlewares** | `src/middlewares/` | Funciones intermedias: verificación de JWT y logging. |
| **Controllers** | `src/controllers/` | Lógica de negocio: validación, consultas SQL y respuestas. |
| **Config** | `src/config/` | Conexión a MySQL mediante un pool reutilizable. |

Este patrón es una variante simplificada de **MVC** (Model-View-Controller) adaptada a una API REST sin vistas. La separación facilita el mantenimiento: añadir un nuevo recurso supone añadir un fichero en `routes/`, otro en `controllers/`, y registrar la ruta en `index.js`.

## 2.3. Flujo de una petición protegida

Tomando como ejemplo `GET /projects`, el flujo completo es:

1. **Cliente** envía la petición con la cabecera `Authorization: Bearer <token>`.
2. **Express** la enruta hacia `projects.routes.js`.
3. Antes de llegar al controlador, el **middleware `authMiddleware`** intercepta la petición:
   - Extrae el token de la cabecera.
   - Lo verifica con `jwt.verify()` usando la clave secreta del servidor.
   - Si es válido, adjunta el payload del token a `req.user` (id, username, email).
   - Si no es válido o falta, responde con `401` o `403` y corta el flujo.
4. El controlador `projects.controller.js` ejecuta la query SQL **filtrando por `req.user.id`**, de modo que la base de datos solo devuelve los proyectos del usuario autenticado.
5. La respuesta se serializa en **JSON** y se devuelve al cliente.

## 2.4. Modelo de datos

El modelo está compuesto por **tres entidades** con **dos relaciones 1:n** entre ellas.

### 2.4.1. Entidades

**Tabla `users`** — representa a los usuarios del sistema:

| Columna | Tipo | Restricciones | Descripción |
|---|---|---|---|
| `id` | INT | PK, AUTO_INCREMENT | Identificador único del usuario |
| `username` | VARCHAR(100) | NOT NULL | Nombre visible del usuario |
| `email` | VARCHAR(150) | NOT NULL, UNIQUE | Email (usado como login) |
| `password` | VARCHAR(255) | NOT NULL | Contraseña hasheada con bcrypt |
| `created_at` | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | Fecha de creación |

**Tabla `projects`** — proyectos que pertenecen a un usuario:

| Columna | Tipo | Restricciones | Descripción |
|---|---|---|---|
| `id` | INT | PK, AUTO_INCREMENT | Identificador único del proyecto |
| `name` | VARCHAR(150) | NOT NULL | Nombre del proyecto |
| `description` | TEXT | — | Descripción libre |
| `user_id` | INT | NOT NULL, FK → `users.id` | Propietario del proyecto |
| `created_at` | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | Fecha de creación |

**Tabla `tasks`** — tareas que pertenecen a un proyecto:

| Columna | Tipo | Restricciones | Descripción |
|---|---|---|---|
| `id` | INT | PK, AUTO_INCREMENT | Identificador único de la tarea |
| `title` | VARCHAR(150) | NOT NULL | Título corto de la tarea |
| `description` | TEXT | — | Descripción libre |
| `completed` | BOOLEAN | DEFAULT FALSE | Indica si la tarea está completada |
| `project_id` | INT | NOT NULL, FK → `projects.id` | Proyecto al que pertenece |
| `created_at` | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | Fecha de creación |

### 2.4.2. Relaciones

| Relación | Cardinalidad | Implementación | Comportamiento al eliminar |
|---|---|---|---|
| `users` → `projects` | **1:n** | `projects.user_id` referencia `users.id` | `ON DELETE CASCADE`: al borrar un usuario, se borran sus proyectos |
| `projects` → `tasks` | **1:n** | `tasks.project_id` referencia `projects.id` | `ON DELETE CASCADE`: al borrar un proyecto, se borran sus tareas |

El comportamiento en cascada garantiza la **integridad referencial** y evita registros huérfanos en la base de datos.

### 2.4.3. Esquema SQL completo

```sql
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(100) NOT NULL,
    email VARCHAR(150) NOT NULL UNIQUE,
    password VARCHAR(255) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE projects (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(150) NOT NULL,
    description TEXT,
    user_id INT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

CREATE TABLE tasks (
    id INT AUTO_INCREMENT PRIMARY KEY,
    title VARCHAR(150) NOT NULL,
    description TEXT,
    completed BOOLEAN DEFAULT FALSE,
    project_id INT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE
);
```

### 2.4.4. Decisiones de diseño relevantes

- **`email UNIQUE`** en la tabla `users` — garantiza que no haya dos cuentas con el mismo correo y permite usarlo como identificador en el login.
- **`password VARCHAR(255)`** — el hash bcrypt ocupa 60 caracteres, pero se reserva más espacio por previsión (otros algoritmos, prefijos largos).
- **`completed BOOLEAN DEFAULT FALSE`** — toda tarea nueva nace como pendiente, simplificando la creación desde el cliente.
- **`created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP`** en todas las tablas — permite ordenar por fecha de creación sin que el cliente tenga que enviar la fecha.
- **No se ha implementado `updated_at`** — el alcance del M1 no lo requería; se podría añadir en futuras iteraciones.

El fichero `mysql/init.sql` carga automáticamente los siguientes datos al inicializar el contenedor de MySQL por primera vez:

- **30 usuarios** (3 destacados como `demo1`, `demo2`, `demo3` para facilitar la presentación).
- **30 proyectos** distribuidos entre los primeros 27 usuarios.
- **30 tareas** distribuidas entre los proyectos, mezclando completadas y pendientes para ilustrar diferentes estados.

Todas las contraseñas en el seed son `demo1234` (hasheadas con bcrypt en formato `$2b$10$...`). Esto cumple holgadamente con el requisito del enunciado de "al menos 30 registros por tabla".

## 2.6. Despliegue con Docker Compose

El fichero `docker-compose.yml` define los dos servicios:

- **`mysql`**: usa la imagen oficial `mysql:8.0`. Monta el script `init.sql` como volumen en `/docker-entrypoint-initdb.d/`, donde MySQL lo ejecuta automáticamente la primera vez que arranca. Los datos se persisten en un volumen llamado `mysql_data`. Expone el puerto 3306 al host como 3307 para evitar conflictos con instalaciones locales de MySQL.

- **`app`**: se construye a partir del `Dockerfile` del repositorio. Depende de `mysql` con la condición `service_healthy`, gracias a un `healthcheck` que comprueba que MySQL responde antes de arrancar Node. Expone el puerto 3000.

Ambos servicios comparten una red `taskmanager-net` y reciben sus credenciales mediante un fichero `.env` (no versionado, plantilla en `.env.example`).
