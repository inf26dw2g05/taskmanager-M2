# Capítulo 3 — Implementación de la API

## 3.1. Visión general de los endpoints

La API expone **15 endpoints** organizados en 4 grupos lógicos. La siguiente tabla resume todos los endpoints, los verbos HTTP utilizados y si requieren autenticación:

| Grupo | Método | Endpoint | Descripción | Auth |
|---|---|---|---|---|
| Auth | POST | `/auth/register` | Registrar un nuevo usuario | No |
| Auth | POST | `/auth/login` | Iniciar sesión y obtener token JWT | No |
| Users | GET | `/users` | Listar todos los usuarios | Sí |
| Users | GET | `/users/me` | Obtener el perfil del usuario autenticado | Sí |
| Users | GET | `/users/:id` | Obtener un usuario por id | Sí |
| Users | PUT | `/users/me` | Actualizar el perfil del usuario autenticado | Sí |
| Users | DELETE | `/users/me` | Eliminar la cuenta del usuario autenticado | Sí |
| Projects | GET | `/projects` | Listar proyectos del usuario autenticado | Sí |
| Projects | GET | `/projects/:id` | Obtener un proyecto por id | Sí |
| Projects | POST | `/projects` | Crear un proyecto | Sí |
| Projects | PUT | `/projects/:id` | Actualizar un proyecto | Sí |
| Projects | DELETE | `/projects/:id` | Eliminar un proyecto | Sí |
| Tasks | GET | `/tasks` | Listar tareas del usuario autenticado | Sí |
| Tasks | GET | `/tasks/:id` | Obtener una tarea por id | Sí |
| Tasks | POST | `/tasks` | Crear una tarea | Sí |
| Tasks | PUT | `/tasks/:id` | Actualizar una tarea | Sí |
| Tasks | DELETE | `/tasks/:id` | Eliminar una tarea | Sí |

Se cumple el requisito del enunciado de utilizar los **cuatro verbos HTTP (GET, POST, PUT, DELETE)** para implementar las operaciones CRUD sobre tres recursos diferentes.

## 3.2. Arranque de la aplicación

El fichero `src/index.js` es el punto de entrada. Su responsabilidad es:

1. Cargar variables de entorno desde `.env`.
2. Inicializar Express y los middlewares globales (`cors`, `express.json`).
3. Inicializar la conexión a MySQL (importando `src/config/db.js`).
4. Montar Swagger UI en `/docs` para servir la documentación OpenAPI.
5. Registrar las rutas de cada grupo, aplicando el middleware de autenticación donde corresponde.
6. Arrancar el servidor en el puerto 3000.

Las rutas se registran de la siguiente forma:

```javascript
app.use("/auth", authRoutes);
app.use("/users", authMiddleware, usersRoutes);
app.use("/projects", authMiddleware, projectsRoutes);
app.use("/tasks", authMiddleware, tasksRoutes);
```

`/auth` es la única ruta pública. El resto pasa por `authMiddleware` antes de llegar al controlador.

## 3.3. Conexión a MySQL: pool de conexiones

El fichero `src/config/db.js` configura un **pool** de conexiones a MySQL en lugar de una conexión única. Esta decisión es importante por tres razones:

1. **Tolerancia a fallos**: si una conexión se cae (por ejemplo, durante el arranque inicial mientras MySQL aún se está inicializando), el pool descarta esa conexión y crea otra automáticamente.
2. **Concurrencia**: múltiples peticiones HTTP pueden ejecutarse en paralelo sin esperar a que la conexión anterior termine.
3. **Estabilidad**: con una conexión única, una desconexión deja la aplicación en estado inutilizable hasta reiniciar. Con un pool, la API se autoregenera.

```javascript
const pool = mysql.createPool({
    host: process.env.DB_HOST,
    user: process.env.DB_USER,
    password: process.env.DB_PASSWORD,
    database: process.env.DB_NAME,
    port: process.env.DB_PORT,
    waitForConnections: true,
    connectionLimit: 10,
    queueLimit: 0
});
```

La API de `pool.query()` es idéntica a la de `connection.query()`, por lo que el cambio fue transparente para los controladores.

## 3.4. Patrón Routes + Controllers

Cada grupo de recursos sigue el mismo patrón: un fichero de rutas que enlaza endpoints con funciones del controlador, y un fichero de controlador con la lógica de negocio.

### Ejemplo: rutas de Projects

```javascript
// src/routes/projects.routes.js
const express = require("express");
const router = express.Router();
const projectsController = require("../controllers/projects.controller");

router.get("/", projectsController.getAllProjects);
router.get("/:id", projectsController.getProjectById);
router.post("/", projectsController.createProject);
router.put("/:id", projectsController.updateProject);
router.delete("/:id", projectsController.deleteProject);

module.exports = router;
```

El router solo se encarga de **mapear**: ningún tipo de lógica de negocio vive aquí. Esto facilita ver de un vistazo qué endpoints expone cada recurso.

### Ejemplo: controlador de Projects

```javascript
// src/controllers/projects.controller.js — extracto
function getAllProjects(req, res) {
    const userId = req.user.id;

    db.query(
        "SELECT * FROM projects WHERE user_id = ?",
        [userId],
        function(err, results) {
            if (err) {
                res.status(500).json(err);
                return;
            }
            res.json(results);
        }
    );
}
```

Observaciones clave:

- **`req.user.id`** proviene del JWT decodificado por el middleware. Está garantizado que existe porque la ruta está protegida.
- **Consultas parametrizadas** (`?` y array de valores) para prevenir inyección SQL. Es lo que recomienda `mysql2` y es práctica obligatoria de seguridad.
- **Filtrado por propietario** directamente en la cláusula `WHERE`: la base de datos solo devuelve los proyectos del usuario que ha hecho la petición.

## 3.5. Operaciones CRUD detalladas

Cada recurso implementa las 5 operaciones típicas. A continuación se describe brevemente cómo se ha implementado cada una para el recurso **Projects** (las otras siguen un patrón análogo).

### 3.5.1. Listar (`GET /projects`)

```sql
SELECT * FROM projects WHERE user_id = ?
```

Devuelve únicamente los proyectos del usuario autenticado. Status `200 OK` con un array JSON.

### 3.5.2. Obtener uno (`GET /projects/:id`)

```sql
SELECT * FROM projects WHERE id = ? AND user_id = ?
```

El doble filtro (`id` y `user_id`) garantiza que un usuario no pueda obtener proyectos ajenos aunque conozca su id. Si la consulta no devuelve resultados, se responde con `404 Not Found` y el mensaje `"Proyecto no encontrado o sin permisos"`.

> **Decisión de diseño**: se devuelve `404` en lugar de `403` deliberadamente. Esto evita que un atacante pueda inferir la existencia de proyectos ajenos mediante la diferencia entre "no existe" (`404`) y "no tienes permiso" (`403`).

### 3.5.3. Crear (`POST /projects`)

```sql
INSERT INTO projects (name, description, user_id) VALUES (?, ?, ?)
```

El `user_id` **no se acepta del cliente**: se toma directamente de `req.user.id`. Esto impide que un usuario malicioso intente crear proyectos en nombre de otro. Status `201 Created` y se devuelve el `projectId` generado.

### 3.5.4. Actualizar (`PUT /projects/:id`)

```sql
UPDATE projects SET name = ?, description = ? WHERE id = ? AND user_id = ?
```

De nuevo, el doble filtro evita la modificación de proyectos ajenos. Si `affectedRows === 0`, se responde con `404`.

### 3.5.5. Eliminar (`DELETE /projects/:id`)

```sql
DELETE FROM projects WHERE id = ? AND user_id = ?
```

Por el `ON DELETE CASCADE` definido en el esquema, eliminar un proyecto elimina también todas sus tareas automáticamente.

## 3.6. Caso especial: tareas anidadas bajo proyectos

El recurso `Tasks` tiene una particularidad: aunque sus endpoints son `/tasks`, las tareas siempre pertenecen a un proyecto. Para garantizar el aislamiento por usuario, las queries hacen `JOIN` con la tabla `projects`:

```sql
SELECT tasks.*
FROM tasks
INNER JOIN projects ON tasks.project_id = projects.id
WHERE projects.user_id = ?
```

Esto significa que un usuario solo verá tareas cuyo proyecto le pertenezca. La verificación se hace **a nivel de base de datos**, no en código Node, lo que es más eficiente y más difícil de eludir.

En la operación de **creación** de tareas se hace una comprobación previa:

```javascript
// Verificar que el project_id indicado pertenece al usuario
db.query(
    "SELECT * FROM projects WHERE id = ? AND user_id = ?",
    [project_id, userId],
    function(err, projects) {
        if (projects.length === 0) {
            res.status(403).json({
                message: "No puedes crear tareas en un proyecto que no es tuyo"
            });
            return;
        }
        // ... INSERT INTO tasks ...
    }
);
```

Si el usuario intenta crear una tarea en un proyecto ajeno, se responde con `403 Forbidden`.

## 3.7. Endpoints `/me` frente a `/:id`

En el recurso `Users` se ha optado por exponer endpoints `/users/me` para las operaciones que afectan al usuario autenticado:

| Endpoint genérico | Endpoint `/me` |
|---|---|
| `PUT /users/:id` (peligroso) | `PUT /users/me` (seguro) |
| `DELETE /users/:id` (peligroso) | `DELETE /users/me` (seguro) |

La razón es que `PUT /users/:id` permitiría a un usuario intentar modificar a otro pasando un id ajeno. Con `PUT /users/me`, el `id` se extrae del token JWT y no del path, por lo que es imposible modificar a un usuario que no sea uno mismo.

`GET /users/:id` se mantiene porque devuelve información pública (sin password) y es razonable que un usuario pueda consultar perfiles ajenos.

> **Detalle de implementación**: en `users.routes.js`, las rutas `/me` se declaran **antes** que `/:id`. Si se hiciera al revés, Express interpretaría la cadena `"me"` como un valor de `:id` numérico, llamaría al controlador `getUserById` y la búsqueda fallaría.

## 3.8. Formato de respuestas

Todas las respuestas son **JSON** (cabecera `Content-Type: application/json; charset=utf-8`). El formato sigue un convenio simple:

- **Éxito con datos**: objeto JSON o array de objetos directamente.
- **Éxito sin datos**: `{ "message": "..." }`.
- **Error**: `{ "message": "...", "error": ... }` con el status code HTTP correspondiente.

### Códigos de status utilizados

| Código | Significado | Casos de uso |
|---|---|---|
| `200 OK` | Operación exitosa | GET, PUT, DELETE |
| `201 Created` | Recurso creado | POST |
| `401 Unauthorized` | Falta token o es inválido | Cualquier endpoint protegido sin auth |
| `403 Forbidden` | Token válido pero sin permiso | Crear tarea en proyecto ajeno |
| `404 Not Found` | Recurso inexistente o sin acceso | GET/PUT/DELETE de id inexistente |
| `500 Internal Server Error` | Error inesperado | Fallos de BD o de servidor |

## 3.9. Documentación con OpenAPI 3.0

La documentación de la API se encuentra en `api/openapi.yaml`, conforme a la especificación **OpenAPI 3.0**. Se sirve a través de **Swagger UI** en el endpoint `/docs`, gracias al paquete `swagger-ui-express` montado en `index.js`:

```javascript
const swaggerDocument = YAML.load("./api/openapi.yaml");
app.use("/docs", swaggerUi.serve, swaggerUi.setup(swaggerDocument));
```

La especificación incluye:

- **Tags** para agrupar endpoints por recurso (Auth, Users, Projects, Tasks).
- **Schemas reutilizables** definidos en `components/schemas` (User, Project, Task, RegisterRequest, etc.).
- **Parámetros reutilizables** (como `IdParam`) en `components/parameters`.
- **Respuestas reutilizables** (`Unauthorized`, `NotFound`, `ServerError`) en `components/responses`.
- **Esquema de seguridad `bearerAuth`** aplicado a nivel global, con sobreescritura en los endpoints públicos (`/auth/register`, `/auth/login`).
- **Ejemplos** de request bodies y de respuestas en cada endpoint.

Swagger UI permite además **probar los endpoints desde el navegador**: tras hacer login y obtener un token, se pulsa el botón "Authorize" para introducirlo y a partir de ahí cualquier endpoint protegido puede ejecutarse desde la propia documentación.

## 3.10. Colección Postman

Se ha preparado una colección Postman (`postman/taskmanager-api.postman_collection.json`) que cubre todos los endpoints organizados en las 4 carpetas. La colección incluye:

- **Variables de entorno** (`baseUrl`, `token`) configuradas a nivel de colección.
- **Autenticación Bearer heredada** desde la raíz de la colección: ninguna petición protegida necesita configurar la cabecera manualmente.
- **Script automático en el endpoint Login**: tras hacer login con éxito, el token JWT recibido se almacena automáticamente en la variable `{{token}}`, de modo que todas las llamadas posteriores lo utilizan sin intervención del usuario.
- **Tests automáticos** en el endpoint Login que verifican el código 200 y la presencia del token en la respuesta.
- **Login alternativo con un segundo usuario** (`demo2`) para demostrar de forma visual el aislamiento por propietario: con un par de clics se puede comprobar que cada usuario ve únicamente sus proyectos y tareas.
